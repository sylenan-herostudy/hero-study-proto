ТЗ портала Hero Study 2.0, разд. 3.3).

1. Контекст и цели

Зачем. В продукте Hero Study есть несколько пользовательских пространств (студент, преподаватель/ППС, менеджер вуза). Цель — встроить единый AI-слой, который постепенно берёт на себя рутину (подсказки, поиск по знаниям, а в перспективе — выполнение действий в интерфейсе) и снимает нагрузку с сотрудников.

Что строим в первую очередь. На текущем этапе важна форма конструкции — «рельсы»: единый, переиспользуемый AI-сервис, к которому потом подключаются конкретные функции. Конкретное наполнение (какой именно сценарий первым) — вторично.

Целевой результат. 1. Единый AI-слой как отдельный переносимый модуль (по образцу модуля IGPA). 2. Сквозной чат (вызов с любой страницы) в личных кабинетах — сначала текст, позже голос. 3. Заложенная возможность выполнять реальные действия в интерфейсе (следующий этап).

Рамочное ограничение (с встречи). Дефицит ресурса; ядро ещё не стабилизировано. Поэтому задача дробится на 1–2 функции до конца года, а AI-слой изолируется, чтобы не влиять на стабильность ядра.

2. Видение и принципы

Принцип Что означает
Агностичность к сущности Слою всё равно, кто обращается (студент / преподаватель / менеджер). Контекст сущности передаётся как параметр.
Агностичность к модели Провайдер меняется конфигом: OpenAI · Anthropic · собственная бесплатная Qwen (self-hosted).
Слой знаний = наша экспертиза Под каждую сущность — своя «база специфических знаний»; именно она отличает продукт от «голой» модели.
Один сервис на всех Единый набор методов, переиспользуемых разными функциональностями, а не отдельный бот под каждый экран.
Отдельный переносимый модуль Чёткий API-контракт и изоляция — как IGPA; модуль легко перенести/выключить, не трогая ядро.
Сначала рельсы, потом функции Сперва каркас и одна точка входа, затем поэтапное наращивание.
Человек в контуре Для действий в интерфейсе — подтверждение пользователем; AI предлагает, человек утверждает.

3. Сущности и базы знаний

Сущность Личный кабинет База специфических знаний (примеры)
Студент ЛК студента Привычки, БД студента, успеваемость, расписание, дедлайны, правила обучения
Преподаватель (ППС) ЛК преподавателя (кабинет ППС — в планах) Функционал ППС, справочник, правила, регламенты, методички, модификаторы
Менеджер вуза ЛК менеджера Регламенты, журнал (посещаемость/оценки), нагрузка, статусы, бизнес-процессы

ППС-специфика подключается после появления кабинета ППС (решение встречи). До этого слой проектируется и пилотируется на студенте и менеджере, но архитектурно поддерживает все три ЛК.

4. Архитектура решения

AI-слой — «чёрный ящик» с единым входом и предсказуемым выходом:

        ┌───────────────────────────────────────────────────────────────┐
        │                    СУЩНОСТИ (точки входа)                      │
        │   ЛК Студента      ЛК Преподавателя(ППС)      ЛК Менеджера      │
        │       │                   │                       │            │
        │       └──── Сквозной чат-виджет (текст; позже — голос) ────────┘
        └────────────────────────────┬──────────────────────────────────┘
                                      │  запрос + контекст сущности (роль, права)
                                      ▼
        ┌───────────────────────────────────────────────────────────────┐
        │                AI-СЛОЙ  (отдельный изолированный модуль)        │
        │                                                               │
        │   ┌─────────────┐    ┌───────────────┐    ┌────────────────┐   │
        │   │ Оркестратор │──▶ │  Слой знаний  │    │  Слой действий │   │
        │   │  (методы:   │    │  (RAG /       │    │  (tools —      │   │
        │   │  chat,      │    │  справочники  │    │  этап 3)       │   │
        │   │  summarize, │    │  по сущности) │    └───────┬────────┘   │
        │   │  extract,   │    └───────┬───────┘            │ вызовы     │
        │   │  act)       │            │ контекст знаний    │ методов UI │
        │   └──────┬──────┘            │                    │ /API ядра  │
        │          │  provider-agnostic интерфейс                       │
        │          ▼                                                    │
        │   ┌──────────────────────────────────────────────────┐       │
        │   │  Провайдер-адаптеры (сменяемы конфигом)            │       │
        │   │  OpenAI  ·  Anthropic  ·  self-hosted Qwen         │       │
        │   └──────────────────────────────────────────────────┘       │
        └────────────────────────────┬──────────────────────────────────┘
                                      │  ответ / выполненное действие
                                      ▼
                       Назад в интерфейс сущности

Компоненты:

  1. Чат-интерфейс (сквозной виджет) — вызывается с любой страницы ЛК. Этап 1 — текст; этап 2 — голосовая транскрибация. Передаёт в слой запрос + контекст сущности (роль и права доступа).
  2. Оркестратор (AI Service) — единый набор методов: chat, summarize, extract, act. Не зависит от конкретного провайдера; собирает промт, подмешивает знания, при необходимости вызывает инструменты.
  3. Слой знаний (Knowledge Base) — RAG/справочники под каждую сущность; «наша экспертиза». Возвращает релевантный контекст под запрос.
  4. Слой действий (Tools/Actions) — этап 3 — выполнение реальных операций в интерфейсе через API ядра (посещаемость, оценки, нагрузка, статусы) с подтверждением пользователем.
  5. Провайдер-адаптеры — реализация единого интерфейса под OpenAI / Anthropic / self-hosted Qwen; смена модели — конфигом, без переписывания функций.

4.1. Схема работы на бэкенде (жизненный цикл запроса)

[ЛК · чат-виджет]
      │  POST /ai/chat   (JWT + контекст сущности)
      ▼
[1) API Gateway] ─▶ [2) Auth / RBAC]    роль · права · вуз (tenant)
      │
      ▼
[3) Оркестратор (AI Service)]
   a) Guardrails / модерация входа
   b) Retrieval ───────────────▶ [Слой базы знаний]  (см. 4.2)
   c) Сборка промта: system + знания + история диалога + запрос
   d) Провайдер-адаптер ───────▶ LLM (OpenAI / Anthropic / Qwen — выбор конфигом)
   e) Действие? ───────────────▶ [Слой действий]: предпросмотр ▶ подтверждение ▶ API ядра
   f) Guardrails / редакция PII на выходе
      │
      ▼
[4) Память диалога · Аудит · Учёт стоимости]        [Кэш: ответы / эмбеддинги]
      │
      ▼
[ЛК] ◀── стрим ответа

По шагам: (1) ЛК шлёт запрос с JWT и контекстом сущности; (2) Gateway → Auth/RBAC определяет роль, права и вуз (tenant); (3) оркестратор: модерация входа → retrieval из БЗ с фильтром по правам → сборка промта → вызов LLM через адаптер выбранной модели (стриминг) → при команде-действии формируется предпросмотр и после подтверждения вызывается API ядра → модерация и редакция PII на выходе; (4) запись истории, аудита и стоимости, кэш ускоряет повторы.

Принципы: stateless-оркестратор (горизонтальное масштабирование), очередь для тяжёлых задач, таймауты/ретраи на адаптерах, полная изоляция модуля от ядра (отдельный сервис/деплой).

4.2. Слой базы знаний (схема)

ИСТОЧНИКИ:   документы (РПД, методички, регламенты) · БД продукта (оценки,
             расписание, нагрузка) · справочники / FAQ
      │
      ▼
INGESTION / ETL:   коннекторы ▶ нормализация ▶ чанкинг ▶ метаданные
                   (tenant, сущность, роль, тип, ACL) ▶ эмбеддинги
      │
      ▼
ХРАНИЛИЩЕ:   vector store (namespace на сущность: студент / преподаватель / менеджер)
             + keyword/BM25 индекс · изоляция по вузу (tenant)
      │
      ▼
RETRIEVAL:   запрос ▶ фильтр прав/ACL (tenant + роль) ▶ hybrid search (vector + BM25)
             ▶ re-rank ▶ top-k ▶ контекст в промт
      ▲
      └── обновление: инкрементальная переиндексация при изменении источников · версионирование
  • Изоляция по вузу (tenant) и сущности (роль): отдельные namespace — студент / преподаватель / менеджер, плюс общий слой (общевузовские регламенты).
  • Безопасность retrieval: на каждый запрос — фильтр по правам роли (ACL); AI «видит» только разрешённое.
  • Гибридный поиск: вектора (смысл) + keyword/BM25 (точные термины) + re-rank.
  • Обновление без переобучения: правка/дозагрузка документов → переиндексация; основной канал «нашей экспертизы» (см. уровень RAG в 4.3).

4.3. Дообучение и настройка (уровни кастомизации)

От дешёвого и быстрого к глубокому — «лестница»; в большинстве случаев достаточно L0–L2:

 глубже / дороже
   ▲   L5  Собственная модель (self-hosted Qwen): обучение на приватных данных on-prem
   │   L4  Fine-tuning / LoRA: дообучение весов на собранных диалогах и разметке
   │   L3  Few-shot: эталонные примеры подмешиваются в промт
   │   L2  База знаний (RAG): дозагрузка документов — «дообучение без обучения»  ◀ основной способ
   │   L1  Промпты и роли: системные промты per сущность / вуз (тон, правила)
   ▼   L0  Конфигурация: модель, параметры (temperature, лимиты), вкл/выкл инструментов
 быстрее / дешевле

 Петля: оценки ответов (👍/👎) ▶ датасет ▶ обновление БЗ (L2) / периодический fine-tune (L4) ▶ лучше
  • Настройки per-tenant (вуз) и per-entity (роль): свои промты, своя БЗ, свой набор инструментов, лимиты.
  • L0–L2 — рабочая зона ближнего скоупа: без обучения весов, безопасно, обновляется на лету.
  • L4–L5 — для своей модели (Qwen): LoRA/QLoRA на приватных данных вуза, on-prem, когда нужна максимальная приватность и специфика.
  • Сбор данных для дообучения: логи диалогов + оценки → размеченный датасет → периодический прогон обновляет БЗ и/или веса.

5. Функциональные требования (по этапам)

Этап 0 — «Рельсы» (MVP-скелет) — ближний скоуп

  • Каркас AI Service с единым интерфейсом методов и минимум одним провайдер-адаптером.
  • Смена модели конфигом (провайдер + ключ + параметры).
  • Подключена одна база знаний (одна сущность) как proof-of-concept.
  • Одна точка входа (чат) на одной странице.
  • Модуль изолирован и переносим; только чтение/советы, без действий.

Этап 1 — Сквозной чат в ЛК (текст) — ближний скоуп, 1–2 функции до конца года

  • Чат-виджет вызывается сквозняком на всех страницах в трёх ЛК (студент, преподаватель, менеджер).
  • Агностичен к сущности; на вход — контекст и права роли.
  • Поверх соответствующей базы знаний (ответы на основе нашей экспертизы, а не общей модели).
  • Подключение конкретного ЛК — по готовности кабинета (преподаватель/ППС — когда появится кабинет).
  • 1–2 рабочие функции на старте (например: вопрос-ответ по регламентам/расписанию, поиск по знаниям).

Этап 2 — Голосовой ввод

  • Транскрибация голосовой команды в текст на входе чата; далее — тот же конвейер.

Этап 3 — Действия в интерфейсе (агентность) — после стабилизации ядра

  • По текстовой/голосовой команде — выполнение реальных операций: «выставь посещаемость», «выставь всем пятёрки» (с ручной правкой остальных), обработка нагрузки, простановка статусов.
  • Подтверждение пользователем перед применением; предпросмотр изменений; возможность отката.
  • Цель — снять рутинную нагрузку с сотрудников.

Отложено

  • ППС-специфика — после появления кабинета ППС.
  • Инструмент AI-навигации по продукту (вытекает из чата).

6. Нефункциональные требования

  • Стабильность/устойчивость — слой ведёт себя как предсказуемый «чёрный ящик»; сбой AI не валит основной интерфейс (graceful degradation, таймауты, ретраи).
  • Изоляция модуля — чёткий API-контракт, отдельное развёртывание; лёгкий перенос/выключение (как IGPA), чтобы не влиять на стабилизацию ядра.
  • Приватность данных студентов и вуза — PII не уходит во внешние модели без основания; для чувствительных данных — опция self-hosted Qwen (on-prem).
  • Безопасность и права — AI работает строго в рамках прав сущности (RBAC): видит и меняет только то, что доступно роли. Авторизация на каждый запрос.
  • Аудит — логирование запросов и (особенно) действий: кто, что, когда, какой результат.
  • Контроль стоимости — лимиты на токены/запросы, кэширование, выбор модели под задачу (дешёвая/локальная для простого, мощная для сложного).
  • Latency/UX — стриминг ответов; индикаторы загрузки; корректные пустые состояния и ошибки.
  • Разворачивание модели на сервере — проработать, как модель «ест» информацию, ресурсные требования self-hosted Qwen, мониторинг и устойчивость под нагрузкой.

8. Риски и ограничения

Риск Митигация
Дефицит ресурса (главный) Дробление на 1–2 функции; начать только с рельс; не раздувать скоуп
Ядро не стабилизировано Полная изоляция AI-модуля; отдельное развёртывание; не трогать ядро
Нет кабинета ППС ППС-этап отложен; архитектура заранее поддерживает 3 сущности

9. Решения и открытые вопросы

Решения (с встречи): рельсы как отдельный переносимый модуль · агностичность к модели и сущности · дробление до 1–2 функций до конца года · ППС — после кабинета ППС · приоритет стабильности ядра.

Открытые вопросы: приоритет первого сценария чата в каждом ЛК · детали разворачивания модели на сервере · сроки кабинета ППС · распределение ресурса и жертвы в roadmap · фидбэк Влады.
